## 1. Element Selection Overlay

### How the Pros Do It

**Chrome DevTools** uses a dedicated overlay layer rendered as an HTML document injected into the inspected page. The overlay receives element geometry via `getBoundingClientRect()` and computed styles, then draws box model visualizations (content, padding, border, margin as colored rectangles) using absolutely-positioned HTML elements. The source lives in Chromium's `front_end/inspector_overlay/` directory — it's plain HTML/CSS/JS, not Canvas.

**Webflow** renders the user's site inside an iframe and overlays selection UI (blue outlines, resize handles, spacing guides) on top. The overlay is a separate layer in the parent document that reads geometry from the iframe's DOM.

**Figma** uses a WebGL canvas for all rendering, which is a different paradigm (full custom rendering engine), but the UX pattern — click to select, blue bounding box, resize handles at corners/midpoints, spacing/margin visualizations — is the gold standard users expect.

**Webstudio** (open source, closest to Edit's architecture) uses an iframe with `postMessage` as the communication layer between Builder and Canvas. State synchronization uses JSON patches generated by Immer.js, distributed via Nanostores.

### Implementation Architecture

The recommended pattern for Edit's iframe + overlay approach:

```
Parent Window (Editor)
├── Toolbar / Panels (React)
├── Selection Overlay (positioned absolutely over iframe)
│   ├── Bounding box outline
│   ├── Resize handles (8 points)
│   ├── Margin visualization (orange)
│   ├── Padding visualization (green)
│   ├── Spacing guides (dimension labels)
│   └── Hover highlight (light blue)
└── iframe (user's site)
    └── Injected script (handles clicks, reports geometry via postMessage)
```

### Key Technical Decisions

**Overlay positioning strategy**: Use `position: fixed` elements in the parent document, positioned using coordinates translated from the iframe's coordinate space. The iframe's `getBoundingClientRect()` gives the iframe's offset in the parent viewport; the element's `getBoundingClientRect()` within the iframe gives position relative to the iframe's viewport. Sum them.

**Coordinate translation formula**:
```typescript
function getOverlayRect(iframeEl: HTMLIFrameElement, targetEl: Element): DOMRect {
  const iframeRect = iframeEl.getBoundingClientRect();
  const elementRect = targetEl.getBoundingClientRect();
  return new DOMRect(
    iframeRect.left + elementRect.left,
    iframeRect.top + elementRect.top,
    elementRect.width,
    elementRect.height
  );
}
```

**Box model visualization**: Use the `css-box-model` npm package (by Alex Reardon, the author of react-beautiful-dnd) rather than raw `getBoundingClientRect()`. It returns separate `marginBox`, `borderBox`, `paddingBox`, and `contentBox` rects from `getComputedStyle()`, which is exactly what you need for DevTools-style box model coloring.

**Handling scroll**: The iframe will scroll independently. Either listen to `scroll` events in the iframe (via the injected script, reporting via `postMessage`) and update overlay positions, or use `requestAnimationFrame` polling to continuously sync. Webstudio uses the RAF approach for smooth tracking.

**Handling zoom/transform**: If the iframe has `transform: scale()` applied (for responsive preview), all coordinates must be multiplied by the scale factor.

### Communication Protocol (Parent <-> iframe)

```typescript
// Messages from iframe to parent
type IframeToParent =
  | { type: 'element-hovered'; path: string; rect: DOMRect; boxModel: BoxModelData }
  | { type: 'element-clicked'; path: string; rect: DOMRect; boxModel: BoxModelData }
  | { type: 'element-deselected' }
  | { type: 'scroll'; scrollX: number; scrollY: number }
  | { type: 'dom-mutated'; mutations: SerializedMutation[] }
  | { type: 'resize'; width: number; height: number };

// Messages from parent to iframe
type ParentToIframe =
  | { type: 'select-element'; path: string }
  | { type: 'highlight-element'; path: string }
  | { type: 'update-style'; path: string; property: string; value: string }
  | { type: 'update-content'; path: string; html: string }
  | { type: 'move-element'; sourcePath: string; targetPath: string; position: 'before' | 'after' | 'inside' }
  | { type: 'enable-edit-mode'; path: string }
  | { type: 'disable-edit-mode' };
```

### Resize Handles Implementation

Eight handles at the four corners and four midpoints of the bounding box. Each handle is a small (8x8px) square element with `cursor` set appropriately (`nw-resize`, `n-resize`, etc.). On drag:

1. Calculate delta from drag start
2. Compute new width/height (corners) or width-only/height-only (midpoints)
3. Send `update-style` messages for `width` and `height` to the iframe
4. Update overlay position in real-time during drag (optimistic, before iframe confirms)

## 2. CSS Style Panel

### How Webflow Organizes CSS Properties

Webflow's style panel is the gold standard for visual CSS editing. Their 2025-2026 redesigned panel breaks CSS into these sections, each collapsible independently or in "Focus mode" (one section at a time):

| Section | CSS Properties | Control Types |
|---------|---------------|---------------|
| **Layout** | `display` (block/flex/grid/inline/none), flex direction, wrap, align, justify, gap | Toggle buttons, icon grids, visual alignment pickers |
| **Spacing** | `margin`, `padding` (all 4 sides, with linked/unlinked toggle) | Visual box model diagram (click each side to edit), number inputs with unit dropdowns |
| **Size** | `width`, `height`, `min-width`, `max-width`, `min-height`, `max-height`, `overflow` | Number inputs with unit dropdowns (px/rem/%/vw/vh/auto/fit-content) |
| **Position** | `position` (static/relative/absolute/fixed/sticky), `top`/`right`/`bottom`/`left`, `z-index`, `float`, `clear` | Toggle buttons, offset inputs, visual position diagram |
| **Typography** | `font-family`, `font-size`, `font-weight`, `line-height`, `letter-spacing`, `text-align`, `text-decoration`, `text-transform`, `color`, `text-shadow` | Font picker, sliders, color picker, alignment buttons |
| **Backgrounds** | `background-color`, `background-image`, gradients, `background-size`, `background-position`, `background-repeat` | Color picker, image uploader, gradient editor, visual position picker |
| **Borders** | `border` (width/style/color per side), `border-radius` (per corner) | Number inputs, style dropdowns, color picker, corner radius visual editor |
| **Effects** | `opacity`, `box-shadow`, `transform` (2D & 3D), `transition`, `filter`, `backdrop-filter`, `cursor`, `mix-blend-mode` | Sliders, shadow editor, transform controls, transition builder |

### Building the Style Panel in React

**There is no off-the-shelf "CSS style panel" npm package** that replicates Webflow's UI. You will build this yourself, but you can compose it from solid primitives:

#### Recommended Component Primitives

| Component Need | Recommended Package | Why |
|---------------|-------------------|-----|
| Number input with scrub (drag to change value) | Custom (see pattern below) | No perfect package exists; build with pointer events |
| Unit selector (px/rem/%/vw) | Custom dropdown | Tightly coupled to the number input |
| Color picker | `react-colorful` (3KB) or `@radix-ui/react-color-picker` | react-colorful is the smallest, most composable option |
| Slider | `@radix-ui/react-slider` | Accessible, composable, unstyled |
| Toggle group | `@radix-ui/react-toggle-group` | For layout direction, text-align, etc. |
| Select/dropdown | `@radix-ui/react-select` | For font-family, font-weight, position, etc. |
| Collapsible sections | `@radix-ui/react-collapsible` | For expandable property groups |
| Tooltip | `@radix-ui/react-tooltip` | Property name hints |

#### Scrub-to-Edit Number Input Pattern

This is the most important custom component. Webflow and Figma both use drag-to-scrub on number inputs:

```typescript
// Core pattern for a scrubbable CSS value input
interface CSSValueInputProps {
  property: string;
  value: number;
  unit: string;
  min?: number;
  max?: number;
  step?: number; // Hold shift for 10x, alt for 0.1x
  onChange: (value: number, unit: string) => void;
}

// Implementation notes:
// 1. On pointerdown on the label, start scrub mode
// 2. On pointermove, calculate delta.x * step, update value
// 3. On pointerup, end scrub mode
// 4. On double-click, switch to text input mode for direct typing
// 5. Support arrow keys (up/down) for increment/decrement
// 6. Shift+drag = 10x multiplier, Alt+drag = 0.1x multiplier
// 7. Use requestPointerLock() for infinite drag range
```

#### Computed vs. Authored Styles

This is a critical design decision:

- **Authored styles**: What the user has explicitly set. Read from the source file or from an in-memory style map. These are what get written back to the CSS file.
- **Computed styles**: What the browser actually applies, including inherited values and defaults. Read via `getComputedStyle()` in the iframe.

**Pattern**: Show authored values as primary (editable). Show computed values as placeholders/hints (grayed out, read-only) for properties the user hasn't set. This is what both Webflow and Chrome DevTools do.

```typescript
interface StylePanelProperty {
  property: string;       // e.g., "font-size"
  authoredValue: string | null;  // e.g., "16px" or null if inherited
  computedValue: string;  // e.g., "16px" (always present)
  isInherited: boolean;   // true if coming from parent
  source: 'inline' | 'class' | 'inherited' | 'default';
}
```

#### Webstudio's Approach to CSS

Webstudio built a JSON-serializable CSS Typed Object Model (inspired by the Houdini CSS Typed OM API) that gives fully typed access to CSS values. This enables bidirectional synchronization between the style panel and the canvas. They also integrated W3C Design Tokens for their variable system.

For Edit's V1, a simpler approach is fine: store styles as `Record<string, string>` per element, and use `getComputedStyle()` as a fallback for unset properties.

## 3. Drag-and-Drop Reordering

### Library Landscape (2026)

| Library | Best For | Bundle Size | Active? | iframe Support |
|---------|----------|-------------|---------|----------------|
| **@dnd-kit** (v1.x, new API) | Visual editors, sortable trees, maximum flexibility | ~10KB core | Yes, actively maintained | No built-in, but buildable |
| **@hello-pangea/dnd** | Simple list reordering | ~35KB | Yes (react-beautiful-dnd fork) | Custom fork needed |
| **pragmatic-drag-and-drop** (Atlassian) | Framework-agnostic, lightweight | ~5KB | Yes | No built-in |
| **formkit/drag-and-drop** | Minimal drag-and-drop | ~5KB | Yes, early stage (v0.3) | No |

### Recommendation: @dnd-kit

**@dnd-kit is the clear winner for visual editors.** Puck (the leading open-source React visual editor) switched from a forked hello-pangea/dnd to dnd-kit specifically for the flexibility needed in visual editor contexts.

Key reasons:
- **Custom collision detection**: Essential for visual editors where drop zones are complex (before/after/inside an element). The `directionBiased`, `pointerIntersection`, and `closestCenter` algorithms can be mixed.
- **No opinion on DOM structure**: Unlike hello-pangea/dnd which requires specific wrapper elements, dnd-kit works with any DOM shape.
- **Tree support**: The `dnd-kit-sortable-tree` package provides tree-based reordering out of the box.
- **Performance**: Designed for pixel-level precision without jank.

### The Cross-iframe Drag Challenge

This is the hardest unsolved problem in the drag-and-drop space. No library natively supports dragging FROM a panel in the parent window INTO an iframe. Here's why it's hard:

1. **Separate DOM trees**: The parent and iframe have independent DOMs. Pointer events in the parent don't fire in the iframe and vice versa.
2. **Coordinate translation**: Drop coordinates must be translated between the two coordinate systems.
3. **Drag preview**: The preview element (drag ghost) needs to visually cross the iframe boundary.

### Cross-iframe Drag Strategy for Edit

**Option A: Overlay-only drag (recommended for V1)**

All drag interactions happen in the overlay layer (parent document). The iframe is passive (pointer-events: none during drag). The overlay draws drop indicators (horizontal lines, blue outlines) based on the hovered element's geometry, which it knows from the communication protocol.

```
1. User drags element in Layers panel or from component palette
2. Parent captures all pointer events (iframe has pointer-events: none)
3. On pointermove, translate coordinates to iframe space
4. Send "get-drop-targets" message to iframe script
5. Iframe responds with valid drop positions + rects
6. Parent draws drop indicators in the overlay
7. On pointerup, send "move-element" command to iframe
8. iframe executes DOM move, parent updates overlay
```

**Option B: Dual-context dnd-kit (advanced)**

Run a `DragDropManager` in both the parent and iframe. Coordinate via postMessage. The parent manager handles drag start/preview; when the pointer enters the iframe rect, hand off to the iframe's manager for precise drop targeting. This is what Puck's approach evolved toward.

**Option C: Use native HTML Drag and Drop API**

The native Drag and Drop API actually crosses iframe boundaries natively (if same-origin). `dragenter`, `dragover`, and `drop` events fire in the iframe even when the drag started in the parent. However, the native API has notorious UX limitations (no custom drag previews, no spring-based animations). Could work for V1 with custom visual feedback layered on top.

### Tree Reordering Pattern

For reordering in a DOM tree (not just flat lists), use `@dnd-kit/react` with the sortable preset and grouping:

```typescript
// Each element is sortable within its parent group
const { ref, isDragging } = useSortable({
  id: element.path,          // unique DOM path identifier
  index: element.siblingIndex,
  group: element.parentPath,  // group by parent element
  type: 'element',
  accept: 'element',
});
```

The `onDragOver` handler detects cross-group transfers (element moved from one parent to another) while `onDragEnd` handles same-group reordering.

## 4. Content Inline Editing

### The contentEditable Landscape

Raw `contentEditable` is widely considered one of the most treacherous browser APIs. Every browser has different behavior for cursor management, keyboard handling, clipboard paste, and undo. Building on raw contentEditable is not recommended.

### Framework Comparison (2026)

| Framework | Bundle Size | React Integration | Best For |
|-----------|------------|------------------|----------|
| **Tiptap** (ProseMirror-based) | ~56KB | First-class | Visual editors, collaborative editing, extensible |
| **Lexical** (Meta) | ~65KB | First-class | Performance-critical, large documents, React Native |
| **Slate.js** | ~45KB | First-class | Maximum control, custom schemas |
| **Quill** | ~400KB | Community wrapper | Legacy; avoid for new projects |

### Recommendation: Tiptap for Rich Text, Custom contentEditable for Simple Text

For Edit's use case, there are two distinct inline editing needs:

**1. Simple text editing (headings, paragraphs, labels)**

A lightweight custom approach is fine for plain text elements. Double-click the element, set `contentEditable=true` on the actual DOM element in the iframe, handle `blur` to save:

```typescript
// In the iframe's injected script
function enableInlineEdit(element: HTMLElement) {
  element.contentEditable = 'true';
  element.focus();

  // Select all text on enter
  const range = document.createRange();
  range.selectNodeContents(element);
  window.getSelection()?.removeAllRanges();
  window.getSelection()?.addRange(range);

  // Save on blur
  element.addEventListener('blur', () => {
    element.contentEditable = 'false';
    postMessage({
      type: 'content-updated',
      path: getElementPath(element),
      html: element.innerHTML,
    });
  }, { once: true });

  // Save on Enter (for single-line elements)
  element.addEventListener('keydown', (e) => {
    if (e.key === 'Enter' && !e.shiftKey) {
      e.preventDefault();
      element.blur();
    }
    if (e.key === 'Escape') {
      element.innerHTML = originalContent; // revert
      element.blur();
    }
  });
}
```

**2. Rich text editing (blog content, descriptions with formatting)**

Embed Tiptap for rich text blocks. Since the editing happens inside the iframe, initialize a Tiptap editor instance within the iframe's injected script:

```typescript
import { Editor } from '@tiptap/core';
import StarterKit from '@tiptap/starter-kit';

function enableRichEdit(element: HTMLElement) {
  const editor = new Editor({
    element,
    extensions: [StarterKit],
    content: element.innerHTML,
    onBlur: ({ editor }) => {
      postMessage({
        type: 'content-updated',
        path: getElementPath(element),
        html: editor.getHTML(),
      });
      editor.destroy();
    },
  });
}
```

Tiptap's core is framework-agnostic (it only requires ProseMirror). The `@tiptap/react` wrapper is for React components, but `@tiptap/core` works in any DOM context, including inside an iframe.

### contentEditable Challenges to Handle

| Challenge | Solution |
|-----------|----------|
| Paste cleaning (Word, Google Docs) | Tiptap handles this; for custom, intercept `paste` event and sanitize with DOMParser |
| Cursor jumping on re-render | Never re-render the contentEditable element from React while editing; let the browser own it |
| IME input (CJK languages) | Use `compositionstart`/`compositionend` events; don't commit until composition ends |
| Undo conflict | Disable browser's native undo (Cmd+Z) inside contentEditable; route to editor's undo system |
| Link/image handling | For V1, only support text. Links and images can be edited through the style panel. |

### Formatting Toolbar

Show a floating toolbar when text is selected inside an inline-editing element. Position it using `@floating-ui/dom` anchored to the selection range's bounding rect. This toolbar provides bold/italic/link buttons and sends `document.execCommand()` or Tiptap commands.

## 5. Responsive Breakpoints

### How Visual Editors Handle This

**Webflow's approach**: The iframe containing the site is rendered at the target viewport width. The editor chrome (panels, toolbar) stays at full desktop width regardless. The iframe is resized by changing its CSS `width`, and a resize handle lets you drag between standard breakpoints. Breakpoint buttons (desktop 992+, tablet 768-991, mobile landscape 480-767, mobile portrait 0-479) snap the iframe width.

**Builder.io**: Uses a dropdown for breakpoints and a draggable resize handle.

**Breakdance/Divi 5**: Similar pattern with a breakpoint bar above the canvas and a draggable handle.

### Implementation Pattern

```
┌──────────────────────────────────────────────────────┐
│  ◻ Desktop  ◻ Tablet  ◻ Mobile  ◻ Custom   [920px]  │  <- Breakpoint bar
├──────────┬───────────────────────────┬───────────────┤
│          │ ┌───────────────────────┐ │               │
│  Layers  │ │                       │ │  Style Panel  │
│  Panel   │ │    iframe             │ │               │
│          │ │    (width: 768px)     │ │               │
│          │ │                       │ │               │
│          │ │                       │ │               │
│          │ └───────────────────────┘ │               │
│          │         ↕ drag handle     │               │
├──────────┴───────────────────────────┴───────────────┤
```

#### Viewport Resize Component

```typescript
interface BreakpointConfig {
  name: string;
  icon: React.ReactNode;
  width: number;        // 0 = responsive/fit
  minWidth?: number;
  maxWidth?: number;
}

const DEFAULT_BREAKPOINTS: BreakpointConfig[] = [
  { name: 'Desktop',          icon: <Monitor />,    width: 0 },     // 0 = fill available
  { name: 'Tablet',           icon: <Tablet />,     width: 768 },
  { name: 'Mobile Landscape', icon: <SmartPhone />, width: 480 },
  { name: 'Mobile Portrait',  icon: <SmartPhone />, width: 375 },
];

// The iframe wrapper:
// - Centers the iframe horizontally when width < available space
// - Adds a subtle drop shadow to show the viewport boundary
// - Uses CSS transition on width for smooth resize animation
// - A draggable handle on the right edge allows free-form resizing
```

#### Considerations

- **Snap guides**: When dragging the handle, snap to common breakpoints (320, 375, 480, 768, 1024, 1280, 1440px) with a visual indicator.
- **Scale to fit**: When the preview width exceeds the available canvas space (e.g., previewing 1440px on a 1200px monitor), apply `transform: scale()` to the iframe and adjust overlay coordinates accordingly.
- **Breakpoint-specific styles**: In V1 (plain HTML/CSS), breakpoints correspond to `@media` queries in the CSS. The style panel should show which media query context is active and allow editing styles for the current breakpoint.
- **Height**: Keep height at 100% of the canvas area (not fixed). Some editors offer a height toggle for "fold" visualization.

## 6. Undo/Redo System

### The Command Pattern (Industry Standard)

Every visual editor uses some variant of the Command Pattern for undo/redo. The two main approaches:

**A. Command objects** (Webflow, Figma style): Each user action is encapsulated as a command with `execute()`, `undo()`, and `redo()` methods.

**B. Patch-based** (Webstudio/Immerhin style): Each user action produces a JSON Patch (RFC 6902). Undo applies the inverse patch; redo re-applies the forward patch. Immer.js generates these patches automatically.

### Recommendation: Patch-Based with Immer (for Edit)

The patch-based approach is superior for Edit because:
1. Edit's core operation is "mutate a DOM/CSS model and write changes back to files." Patches naturally represent these diffs.
2. Patches are serializable — they can be sent to the server for file writes and to other clients for collaboration (future).
3. Immer makes it trivial to produce patches from imperative-style mutations.
4. Webstudio has already validated this pattern in production with their `immerhin` library.

### Architecture

```typescript
import { produce, enablePatches, Patch } from 'immer';

enablePatches();

interface Transaction {
  id: string;
  label: string;              // Human-readable: "Changed font-size to 16px"
  timestamp: number;
  patches: Patch[];           // Forward patches
  inversePatches: Patch[];    // Reverse patches (for undo)
}

class UndoManager {
  private undoStack: Transaction[] = [];
  private redoStack: Transaction[] = [];
  private maxHistory = 100;

  execute<T>(state: T, label: string, mutator: (draft: T) => void): T {
    let patches: Patch[] = [];
    let inversePatches: Patch[] = [];

    const nextState = produce(state, mutator, (p, ip) => {
      patches = p;
      inversePatches = ip;
    });

    this.undoStack.push({
      id: crypto.randomUUID(),
      label,
      timestamp: Date.now(),
      patches,
      inversePatches,
    });

    // Clear redo stack on new action (standard behavior)
    this.redoStack = [];

    // Trim history
    if (this.undoStack.length > this.maxHistory) {
      this.undoStack.shift();
    }

    return nextState;
  }

  undo<T>(state: T): T {
    const tx = this.undoStack.pop();
    if (!tx) return state;
    this.redoStack.push(tx);
    return applyPatches(state, tx.inversePatches);
  }

  redo<T>(state: T): T {
    const tx = this.redoStack.pop();
    if (!tx) return state;
    this.undoStack.push(tx);
    return applyPatches(state, tx.patches);
  }

  get canUndo() { return this.undoStack.length > 0; }
  get canRedo() { return this.redoStack.length > 0; }
}
```

### Transaction Batching

Some user actions produce many individual changes that should undo as one unit:
- Dragging a slider (continuous value changes should collapse into one undo step)
- Moving an element (multiple style changes: left, top)
- Search-and-replace (many text changes at once)

**Pattern**: Use a debounced transaction. Start a transaction on the first change; accumulate subsequent changes within a time window (e.g., 300ms). Commit the batch when the window closes.

```typescript
// Batch rapid changes into a single undo step
class BatchTransaction {
  private timer: number | null = null;
  private patches: Patch[] = [];
  private inversePatches: Patch[] = [];
  private batchMs = 300;

  addChange(patches: Patch[], inversePatches: Patch[]) {
    this.patches.push(...patches);
    this.inversePatches.push(...inversePatches);

    if (this.timer) clearTimeout(this.timer);
    this.timer = window.setTimeout(() => this.commit(), this.batchMs);
  }

  private commit() {
    // Push accumulated patches as a single Transaction
    undoManager.pushTransaction({
      patches: this.patches,
      inversePatches: this.inversePatches,
    });
    this.patches = [];
    this.inversePatches = [];
  }
}
```

### Integration with Source File Layer

The undo/redo system operates on the in-memory editor state. The file write layer is a separate concern:

```
User action → UndoManager (patches) → In-memory state update → iframe DOM update
                                     → File writer (debounced, writes patches to source files)
```

File writes should be debounced (e.g., 1s after last change) and batched. The undo system does NOT undo file writes — it undoes in-memory state. File writes are an eventual consistency layer.

### Existing Libraries

| Library | Approach | Notes |
|---------|----------|-------|
| `immerhin` (Webstudio) | Immer patches + Nanostores | Battle-tested in Webstudio; small, purpose-built |
| `redux-undo` | Redux state snapshots | Heavy; only makes sense if using Redux |
| `use-undo` | React hook, state snapshots | Too simple for a visual editor |

**Recommendation**: Use `immerhin` if adopting Nanostores, or build a custom Immer-based system (as shown above) if using Zustand or another state manager.

## 7. Layers/Tree Panel

### Reference Implementations

- **Webflow Navigator**: Shows the full element tree, supports drag-to-reorder, click-to-select (synced with canvas), rename, and visual indicators for element types.
- **Chrome DevTools Elements**: Shows full DOM tree with expand/collapse, search, breadcrumb trail. Hover highlights elements on page.
- **Figma Layers**: Flat-ish layer list with nesting, drag-to-reorder, visibility toggles, lock toggles.
- **VS Code Explorer**: Tree with virtual scrolling, keyboard navigation, multi-select.

### Recommended Library: react-arborist

`react-arborist` (npm: `react-arborist`, 3.4.x, MIT) is purpose-built for exactly this use case. It describes itself as providing "the equivalent of a VSCode sidebar, Mac Finder, Sketch/Figma layers panel."

**Key features**:
- Built-in virtualization (only renders visible rows) — handles thousands of DOM nodes
- Drag-and-drop reordering within the tree
- Inline rename (double-click to edit)
- Multi-select (shift-click, cmd-click)
- Keyboard navigation (arrow keys, enter to rename)
- Controlled or uncontrolled modes
- Fully customizable node rendering

```typescript
import { Tree, NodeRendererProps } from 'react-arborist';

interface DOMTreeNode {
  id: string;        // CSS selector path or data-edit-loc
  name: string;      // Tag name + class, e.g., "div.hero-section"
  children?: DOMTreeNode[];
  elementType: string; // "div", "section", "img", "p", etc.
}

function LayersPanel({ treeData, selectedId, onSelect }) {
  return (
    <Tree
      data={treeData}
      selection={selectedId}
      onSelect={(nodes) => onSelect(nodes[0]?.id)}
      width={280}
      height={600}    // Virtualizes rows; can handle 10,000+ nodes
      rowHeight={28}
      indent={16}
      openByDefault={false}
    >
      {ElementNode}
    </Tree>
  );
}

function ElementNode({ node, style, dragHandle }: NodeRendererProps<DOMTreeNode>) {
  return (
    <div
      style={style}
      ref={dragHandle}
      className={cn('layer-node', { selected: node.isSelected })}
      onClick={() => node.select()}
    >
      <ElementTypeIcon type={node.data.elementType} />
      <span>{node.data.name}</span>
    </div>
  );
}
```

### Alternative: TanStack Virtual + Custom Tree

If you need more control than react-arborist provides (e.g., custom drag behavior that integrates with your dnd-kit setup), build a custom tree on top of `@tanstack/react-virtual`:

```typescript
import { useVirtualizer } from '@tanstack/react-virtual';

// Flatten tree to list (visible nodes only, respecting expand/collapse)
function flattenTree(nodes: TreeNode[], expanded: Set<string>): FlatNode[] {
  const result: FlatNode[] = [];
  function walk(nodes: TreeNode[], depth: number) {
    for (const node of nodes) {
      result.push({ ...node, depth });
      if (expanded.has(node.id) && node.children) {
        walk(node.children, depth + 1);
      }
    }
  }
  walk(nodes, 0);
  return result;
}

// Then use useVirtualizer on the flattened list
const virtualizer = useVirtualizer({
  count: flatNodes.length,
  getScrollElement: () => scrollRef.current,
  estimateSize: () => 28,
  overscan: 20,
});
```

### Selection Synchronization

The tree panel and canvas overlay must stay in sync bidirectionally:

- **Canvas -> Tree**: When user clicks an element on the canvas, the tree scrolls to and highlights the corresponding node.
- **Tree -> Canvas**: When user clicks a node in the tree, the canvas scrolls to and highlights the corresponding element.

```typescript
// Sync hook
function useSelectionSync() {
  const [selectedPath, setSelectedPath] = useState<string | null>(null);

  // From canvas (iframe postMessage)
  useEffect(() => {
    const handler = (e: MessageEvent) => {
      if (e.data.type === 'element-clicked') {
        setSelectedPath(e.data.path);
        treeRef.current?.scrollTo(e.data.path); // react-arborist scroll API
      }
    };
    window.addEventListener('message', handler);
    return () => window.removeEventListener('message', handler);
  }, []);

  // From tree (click handler)
  const handleTreeSelect = (path: string) => {
    setSelectedPath(path);
    iframeRef.current?.contentWindow?.postMessage(
      { type: 'select-element', path },
      '*'
    );
  };

  return { selectedPath, handleTreeSelect };
}
```

### Tree Node Display Conventions

Follow Webflow/DevTools conventions:
- Show tag name in a colored badge (similar to DevTools)
- Show the most meaningful identifier: `id` > first `class` > tag name
- Show element type icon (text, image, container, link, form, etc.)
- Dim nodes that are hidden (`display: none` or `visibility: hidden`)
- Show indicator for elements with CSS classes
- Breadcrumb trail at the bottom showing the path from root to selected element
